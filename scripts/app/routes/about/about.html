About

Overview

This app has been designed to provide the barebones skeleton you need to start writing a scalable, well-architected AngularJS 1.x app with the minimum of configuration hassle, and the minimum of boilerplate code getting in your way.

Most boilerplate templates include a tonne of other stuff, such as old browser upgrade notices, viewport definitions, favicons, Google Analytics, etc., etc. All of this is needed for production code, but it gets in your way if all you want is a clean template to begin a new project with, to create a prototype, or just to experiment with AngularJS without irrelevant HTML5 cruft getting in your way.

Equally, other Angular Seed apps may provide an appropriate place for you to start, but provide you with limited build tools. There's no minification or concatenation, and so you have to add manually all the files you create both to your index.html file, and your karma.conf file.

To this end, this seed app is designed to get you up and running fast to explore AngularJS with the minimum of boilerplate so you can really see the wood from the trees. You can prototype code quickly and easily, with minifying, concatenating, source maps, and testing configuration all taken care of for you.

Main Features

Scaffolding HTML page (i.e index.html)

Index.html is the main HTML scaffolding page. It contains only a custom header directive to get you started with directives, an ng-view directive in which your main content will be rendered via your Angular Router, and a static footer.

Styling is minimal, and is provided by Bootstrap.css, and a small bit of custom CSS declared in main.css, which keeps the footer sticking to the bottom of the page (see CSS-Tricks for more details).

App structure

The structure of the app has been written according to the inestimable John Papa's style guide, with the main app folder comprising the following components:

  • the main appModule (config, run, constants, routes, etc,)
  • layout components (header, footer and sidebar directives)
  • routes (HTML templates and controllers needed for specific routes)
Each folder contains all files necessary for the component to render, run and be tested.

Code Structure

The AngularJS code also complies with John's style recommendations, with controllerAs used throughout, IIFE's enclosing component declarations to avoid globals, and the ViewModel represented as a capture variable called vm.

Building the app

gulp build

Before running the app, you MUST call the build task from the command line (note: I'm assuming you're not using Windows with these instructions).

Once built, you shouldn't need to rebuild unless you add a new file. Editing existing files will cause the app to be rebuilt automatically.

The build task concatenates the files and minifies them, with the third party libraries built to dist/lib.min.js, and your app's code built to dist/app.min.js. The build task also creates sourcemaps, ensuring you can still debug the actual code, rather than an horrendous minified concatenated soup!

The build task also looks for any AngularJS directive templates that have been defined using templateUrl. These are concatenated and written to a module, the contents of which are injected into $templateCache for you. The benefits of this are that a single HTTP request need only be made regardless of how many directives you create, and, more importantly, your directive tests can use the actual partial HTML you use in your production code.

Just include the partials module in your tests, like so:

beforeEach(module('partials'));

and your directives will be magically populated with their proper partials' HTML (see Untangled.io's 'How to Unit Test a directive with templateURL' for more details). You can see an example of this in action in scripts/app/layout/header/header.directive.spec.js

Running the app

gulp serve

Once run, browse to 127.0.0.1:8080. To change the port, just type gulp serve --port=< your_port_number >. For example, to serve the app on port 8000, use gulp serve --port=8000

Running gulp serve causes a live reload, so all your app's files are automatically rebuilt (the library files aren't - if you want those rebuilt, you'll need to callgulp build again).

Testing the app

karma start

This runs the tests in Chrome. To see the output of the tests, click the 'debug' button in the page that's opened, and then open Chrome dev tools. You'll see the output of your tests in the console, and can also step through them using the standard dev tools debugging commands.
